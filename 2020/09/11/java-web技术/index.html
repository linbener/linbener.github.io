    深入WEB请求过程

    如何发起一个HTTP请求？

    发起一个HTTP请求，可以理解成为建立一个Socket连接通信。

    浏览器根据URL地址，通过DNS解析IP地址，建立套字连接，套字根据HTTP协议，向服务器发送请求头数据，然后服务器根据套字读取的返回信息，断开套字。

    HttpClient就是一个开源的通过程序实现的处理HTTP请求的工具包。

    Linux系统的curl工具

    解析HTTP

    要理解HTTP，最重要的事HTTP中的HTTP Header，HTTP Header控制着用户的数据传输，和浏览器的渲染行为，服务器的执行逻辑。

    浏览器缓存

    如果想要浏览器提供的页面不是缓存的，那么需要重新加载页面，而加载页面后，URL的请求头将会增加两个参数

    Cache-Control:no-cache ; Pragma:no-cache

    其他的请求参数

    Expires, 这个请求头会有一个日期，超过了日期，缓存的内容就会失效，这时浏览器就需要重新加载页面

    Last-Modified /Etag， 一般用于表示一个服务器上的资源最后修改时间。Servlet提供一个方法getLastModified方法用于检查某个动态内容是否已经更新。

    DNS解析

    win，Linux下可以通过nslookup命令查询域名解析的结果，Linux还可以通过dig命令

    域名解析记录主要分为 ：A记录 ，地址记录；MX记录，Mail Exchange；CNAME记录，别名解析；NS记录，为某个域名指定DNS服务器；TXT记录。

    CDN工作机制(Content delivery NetWork)

    内容分布网络，它是构筑在现有的internet上的一种先进的流量分配网络。目的是在internet中增加一层新的网络架构，可以将网站的内容发布给最近的用户的网络“边缘”。

    CDN缓存的是静态内容

    用户发起请求时，DNS服务器通过解析把CDN节点IP信息返回给浏览器，浏览器根据IP信息向CDN节点请求静态资源，如果CDN上没有静态资源，那么CDN节点就通过回源，向应用服务器的数据库上请求资源，然后返回给用户。

    CDN动态加速技术：在CDN解析DNS中通过 动态的链路探测来寻找 回源 最好的一条路径，然后将请求调度到这条路径上。

    深入分析Java web中文编码的问题

    常见的码：

    ASCII码：一共有128个，用一个字节的低7位表示，0-31是控制字符，32-126是打印字符。

    ISO-8859-1：依然是单字节编码，一共有256个字符，覆盖大多数西欧语言

    GB2312：双字节编码，总范围是A1-F7，其中A1-A9是符号区，B0-F7是汉字区。

    GBK：扩展了GB2312，编码范围是8140-FEFE，去掉XX7E，一共有23940个码位，能表示21003个汉字

    UTF-16：具体定义了Unicode字符在计算机的存取方法，用2个字节表示。

    UTF-8：不同的字符可以用1-6个字节编码

    Java中的Charset字符集

    Java中进行编码的核心步骤：根据字符集charset，创建CharsetEncoder对象，调用encode方法进行编码。

    UTF-16采用顺序编码，不能对单个字符进行校验，如果中间字节流损坏了，后面的编码将受到影响，而UTF8不存在这些问题。

    UTF-16编码适合在Java的内存和磁盘之间的传输，因为网络容易损坏字节流，UTF-8适合在网络传输

    UTF-8是理想的中文编码

    Java web设计的编码

    URL中的编码

    根据URL规范，URL编码是将URL中的非ASCLL字符按照某种编码方式 编码成16进制数字后，再增加一个%号。

    比如

    /server/林?age=12岁 —- > /server/%AhshJ189?age=%sahjsajsh779

    HTTP中的请求参数编码默认采用ISO，可以通过设置请求头ContentType中设置charset字符集

    同时如果想要调用request.getParamter方法，那么你必须要先设计字符集，否则会出现乱码，request.setCharacterEncoding

    提交表单时的编码，因为表单内容是嵌在HTTP的BODY里面的，所以可以通过ContentType设置编码，

    在JS中的编码问题

    JS文件的编码和 HTML页面的编码 不同，就有可能出现乱码。

    JS发起异步请求的URL编码是受到浏览器的默认编码影响。

    JS处理URL编码的三个函数

    escape() 将ASCII字母，数字，标点符号之外的其他字符转换成Unicode编码值，并且在编码值前加上%

    encodeURI() 真正的JS对URL编码的函数，除某些字符外，对URL进行UTF-8编码，每个码值前加上%

    encodeURIComponet() 比前面的还要更具彻底编码，通常用于将一个URL当作参数放在另外一个URL中

    Java和JS编解码问题

    Java中的类是URLEncoder ,URLDecoder

    这个两个类都可以将所有的%加UTF8的码值用UTF8解码，得到原始字符

    与JS前端对于的函数是encodeURIComponet() decodeURIComponet()

    国际化问题

    一般要将网站的编码格式换成支持多种语言的UTF8

    然后对页面进行本地化翻译，本地化翻译分为：1 机械自动翻译 2 人工翻译

    serlvet工作原理解析

    用tomcat分析

    首先明确一个概念: serlvet 和 serlvet 容器是两个独立的概念。serlvet 是一个类，它在容器中被包装成另一个类，Wrapper。

    serlvet 容器的启动过程

    • Engine:代表整个Catalina servle引擎
    • Host:代表虚拟主机
    • Context:代表某个web应用
    • Wrapper:代表某个应用中的servlet

    tomcat中容器分为4个等级，从父容器到子容器：Container容器 –>Engine容器 –> Host 容器–>Serlvet容器

    而在serlvet容器中，实际管理serlvet的是Context容器，一个Context容器代表一个web工程，而Context容器管理着许多的serlvet类的包装类Wrapper。

    tomcat7, 开始支持嵌入式功能，新增了一个启动类org.apache.catalina.startup.Tomcat

    这个类的对象方法start可以很轻易启动tomcat，并且配置一个WEB工程。

    Tomcat的启动逻辑是基于观察者模式设计的，所有容器都继承了Lifecycle接口，这个接口管理者容器的生命周期，所有容器的修改和状态的编号都会由它通知已经注册的观察者Listener。

    • Lifecycle:相当于抽象主题角色，所有的容器类与组件实现类都实现了这个接口。如StandardContext
    • LifecycleListener:相当于抽象观察者角色,具体的实现类有ContextConfig, HostConfig, EngineConfig类，它们在容器启动时与停止时触发。
    • LifecycleEvent:生命周期事件，对主题与发生的事件进行封装。
    • LifecycleSupport:生命周期管理的实用类，提供对观察者的添加，删除及通知观察者的方法。
    • LifecycleException:生命周期异常类。

    最先启动的是容器

    ContextConfig是一个观察者，它负责整个web应用的配置文件的解析工作。

    它会执行init方法, 然后Context容器会执行startInternet方法

    web应用的初始化工作

    在ContextConfig 中的configureStart方法实现，应用的初始化主要是解析web.xml文件。

    tomcat首先会找globalWebXml，hostWebXml, web.xml 。

    web.xml文件会解析成一个WebXml对象，而这个WebXml对象中的属性会设置到Context容器中，这时包括创建Serlvet,filter,listener等。

    而这其中Serlvet 将会包装成context容器中的StandardWrapper类对象，Serlvet作为一个独立的web开发标准，不应该强耦合在tomcat中，而StandardWrapper作为tomcat容器的一部分，具有容器的特征。

    创建Serlvet实例

    前面的操作中，仅仅是将serlvet包装成另外的类，还没有创建具体的实例

    如果Serlvet的load-on-startup属性配置项大于0，那么context容器启动时将会被创建实例，而最先实例化的Serlvet是在globalWebXml文件中定义的 DefaultSerlvet 和 JspSerlvet，因为他们的load-on-startup大于0.

    创建serlvet实例的方法是从Wrapper.loadSerlvet开始的。

    Serlvet的体系结构

    用户从浏览器向服务器发起一个请求

    hostname:port/contextpath/serlvetpath

    其中hostname,port和服务器上的tomcat建立TCP连接，而后面的contextpath/serlvetpath决定了服务器中的哪个子容器服务用户的请求。

    1 serlvet工作

    在tomcat7中，将有一个映射mapper，这个类保存了tomcat的Container容器中的所有子容器信息，在请求类Request进入Container容器之前，mapper会根据请求URL信息，将host,context容器设置到request的mappingData属性中，所有当request进入容器之前，对于它访问那个子容器就已经确定了。

    这时serlvet就会执行它的service方法，移出容器时执行destroy方法。

    2 serlvet中的listener

    它是基于观察者设计的，它是观察者接口。

    3 serlvet 中的 filter

    Filter 除了提供request,response 外，还提供了 一个 FilterChain 对象，FilterChain代表了当前的整个请求链，它的设计模式是在责任链模式，如果不想把请求继续传递下去，那么就不调用FilterChain的doFilter方法。请求直接会返回，不会抵达最终的serlvet

    4 serlvet 中的 url-pattern

    serlvet中会在mapper类中匹配

    filter会在创建ApplicationFilterChain对象进行

    cookie的属性项

    expires ,domain 域名；path 路径 ；max-age 最大失效时间

    port 该cookie在什么端口下可以回传服务器

    comment 注解cookie的用途

    cookie如何工作

    request.getCookies();

    每次调用response.addCookie，这些cookie对单独的以Header存在，

    理解session

    客户端第一次访问服务器时，服务器生产一个ID，这个ID通常是NANE为JSESIONID的一个cookie。

    每次客户端都会传回给服务器。

    Session基于cookie工作，有三种方式可以让session正常工作：

    1. 基于URL Path Parameter,默认支持
    2. 基于Cookie，如果没有修改context容器的cookie标识，默认支持
    3. 基于SSL，默认不支持，开启connector.getAttribute(“SSLEnabled”)的TRUE时才支持。

    第一种情况，如果客户端不支持cookie，那么浏览器会将用户的SessionCookieName重写到用户请求的URL参数中，传递格式如

    path/serlvet;name=value;name2=value2?name3=kk

    SessionCookieName就写在serlvet:后面，？前面。

    用户可以配置SessionCookieName，在web.xml 文件的属性项session-config，name的属性值就是SessionCookieName的值。

    默认的SessionCookieName是JSESIONID。

    session如何工作

    有了session ID,服务端就可以创建 HttpSession对象了，而这个对象在创建后，会保存在Manager类中。

    第一次触发是调用request.getSession方法，如果没有对象，则会创建HttpSession对象。

    Manager的实现类是 StandardManager，它负责管理整个session的生命周期，它的unload方法会将session集中的StandardSession对象写到“SESSIONS.ser” 文件中持久化

    分布式Session架构, 多终端Session统一

    Tomcat体系结构

    Connector 负责对外交流，这个模块可以更换

    Container 容器，可以拥有多个Connector

    Service是Connector和 Container 的包装，他提供对外的一个服务，一个Seivice只能有一个 Container 。

    Lifecycle接口，可以控制所有组件的生命周期，组件一层一层地向上传递，而在所有组件中，最高级的组件是Service，Service组件可以控制整个Tomcat的启动和停止

    Connector最重要的功能是接收浏览器的TCP请求，并把分配线程让Container处理请求。

    这必然是多线程的，而 Connector的多线程设计是核心

    Container 接口，继承它的有4个容器，最简单的web应用可以只有Context,Wrapper容器，

    而Engine代表了一个完整的Serlvet引擎，Host 代表一个虚拟主机，作用就是允许多个应用，负责安装和展开应用，并且标识这个应用一边能够区分它们，它不是必须的，但是如果要运行war程序，那么它是必须的。

    Tomcat5前，Context找到正确的serlvet处理请求是通过mapper类，但是在5之后，这个功能被移植到了Request中。

    Jetty工作原理

    一个非常重要的数据模型Handler，所有的可以被扩展的组件都可以作为一个handler添加到Server中。

    基本组件是Server 和 Connector

    整个Server组件是基于Handler容器工作的

    HandlerWrapper 可以将Handler委托给另外一个类执行

    ScopeHandler 可以拦截Handler，在调用Handler之前，之后做额外的事情

    HandlerCollection 将多个Handler类组装在一起，构成一个Handler链，方便扩展

    Jetty的入口是Server类，它启动了，就代表jetty能提供服务

    jetty基于两种协议工作，一种是HTTP,一种是AJP模式

    AJP模式

    AJP模式，在企业用户中，Java应用服务器一般不会直接对外提供服务，把应用服务器暴露给服务访问者，而是在Java应用服务器外面，套上一层WEB服务器。

    Java应用服务器一般运行的软件是tomcat,jetty ，而web服务器运行的是apache,nginx。

    如果采用AJP模式工作，那么前面的WEB服务器已经完成了HTTP的解析工作，同时也会完成HTTP的封装工作。

    如果采用AJP模式工作，那么jetty作为一个serlvet引擎，通常会可JBoss集成工作。

    Velocity工作原理

    Java语言中模板引擎使用最多的是JSP, Velocity, FreeMarker

    Velocity的主要组件时runtime app context util

    app组件包装了对外暴露给使用者使用的接口，主要是两个类 Velocity 和 VelocityEngine。

    前者封装了静态接口，可以直接调用，帮助你渲染模板；后者主要是给框架开发者调用的，提供更加复杂的接口。

    大浏览系统的静态架构设计

    静态化系统：

    一个页面对应的URL通常是固定的。

    在页面中不能包含与浏览者相关的因素，即HTML页面不能明显地含有与浏览器相关的dom

    在页面中不含时间因素，这个时间是服务端输出的时间

    页面不包含地域因素

    不包含cookie等私有数据

    页面即HTML源码

    优点：

    改变了缓存方式，直接缓存HTTP连接而不是仅仅缓存数据，Web代理服务器根据请求URL直接取出对应的HTTP响应头和响应体直接返回，不经过Java应用服务器

    改变了缓存的地方。不是在Java层面做缓存，而是直接在web服务器层上做，屏蔽了Java层面的一些弱点。
